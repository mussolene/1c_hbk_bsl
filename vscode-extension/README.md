# BSL Analyzer — расширение для VS Code / Cursor

Расширение подключает [LSP-сервер `bsl-analyzer`](https://github.com/mussolene/1c_hbk_bsl) к редактору: подсветка синтаксиса BSL, диагностики, переход к определению, hover, дополнения, форматирование, переименование и семантические токены для кода **1С:Предприятие** (модули `.bsl`, OneScript `.os`).

**Лицензия:** см. [LICENSE](./LICENSE) (MIT).

---

## Возможности

| Возможность | Описание |
|-------------|----------|
| Диагностики | Правила BSL001… (совместимы по именам с BSL Language Server там, где задано) |
| Навигация | Переход к определению, поиск символов, вызовы |
| Форматирование | Встроенный форматтер, в т.ч. format on save для `[bsl]` |
| Inlay hints | Имена параметров на вызовах |
| Семантика | Дополнительная подсветка поверх TextMate |

Python в рантайме **не нужен** — в работе участвует только нативный бинарник `bsl-analyzer` (PyInstaller и т.п.).

---

## Откуда берётся сервер (`bsl-analyzer`)

Порядок поиска бинарника (как в коде расширения):

1. **`bslAnalyzer.serverPath`** — явный путь к исполняемому файлу (удобно для разработки, `pip install` / локальная сборка — **системный `PATH` не просматривается**).
2. **`bin/` внутри расширения** — если VSIX собран с вложенным бинарником под вашу платформу (релизы GitHub Actions делают именно так).
3. **Кэш после скачивания** — глобальное хранилище расширения.
4. **Предложение скачать** с GitHub Releases — если ничего не найдено (см. константу тега релиза в `src/extension.ts`).

Имеет смысл держать **`serverPath`** пустым или значением по умолчанию `bsl-analyzer`, если вы используете готовый VSIX с бинарником в `bin/` или соглашаетесь на скачивание. Чтобы использовать бинарник из установки в окружение Python, укажите полный путь в **`serverPath`**.

## Упаковывать ли бинарник в VSIX?

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| **Бинарник внутри VSIX** (по платформам: `darwin-arm64`, `linux-x64`, …) | Работает сразу после установки, без сети и без настройки; предсказуемая версия сервера. | Нужно собирать отдельный VSIX на целевую ОС; размер пакета больше. |
| **Только скачивание с Releases** | Меньший универсальный пакет; можно всегда тянуть последний билд с релиза. | Первый запуск требует сети; возможны лимиты GitHub; версия должна совпадать с расширением. |

**Практическая рекомендация:** для публикации в Marketplace оставить **платформенные VSIX с вложенным бинарником** (как сейчас в CI) — лучший UX. Скачивание с релиза — запасной путь и для пользователей без бинарника в пакете. Оба варианта в коде уже поддерживаются; приоритет у явного пути и вложенного `bin/`.

## Настройки

Основные ключи (полный список — в `package.json` → `contributes.configuration`):

| Ключ | Назначение |
|------|------------|
| `bslAnalyzer.serverPath` | Путь к `bsl-analyzer` или `bsl-analyzer.exe` |
| `bslAnalyzer.indexDbPath` | Путь к SQLite-индексу (пусто = по умолчанию) |
| `bslAnalyzer.logLevel` | Уровень логов LSP |
| `bslAnalyzer.diagnostics.enabled` | Включить диагностики при вводе |
| `bslAnalyzer.diagnostics.select` / `ignore` | Список кодов правил |
| `bslAnalyzer.format.indentSize` | Отступ при форматировании |

## Сборка расширения (разработчики)

Из каталога `vscode-extension/`:

```bash
npm ci
npm run compile    # webpack → out/extension.js
```

Упаковка VSIX (локально):

```bash
npx vsce package --no-dependencies -o bsl-analyzer-local.vsix
```

Проверка типов без бандла:

```bash
npx tsc --noEmit -p ./
```
