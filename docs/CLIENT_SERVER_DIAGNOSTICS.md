# Клиент/серверный контекст в диагностиках (партии правил)

В 1С один и тот же файл модуля может содержать методы с разными директивами компилятора (`&НаКлиенте`, `&НаСервере`, `&НаКлиентеНаСервере`). Для правил, где BSLLS зависит от границы клиент/сервер, **не используем** привязку к пути вида `.../CommonCommands/...` — вместо этого:

- контекст метода: `_procedure_compiler_execution_context` → `client` | `server` | `both` | `none`;
- типовой обработчик команды: `_is_typical_client_command_handler` — `Процедура ОбработкаКоманды` в контексте `client` / `both` / `none` (без «только сервер»).

## Партия 1 (реализовано)

| Код | Смысл | Было | Стало |
|-----|--------|------|--------|
| BSL022 | Устаревшее `Предупреждение()` / `Warning()` | отключение по пути CommonCommands | пропуск только в типовом клиентском `ОбработкаКоманды` |
| BSL041 | `ОписаниеОповещения` | то же | то же |
| BSL062 | неиспользуемый параметр `Параметры` / `parameters` | то же | то же + колбэк `*Завершение` / `*Completion` на клиенте |
| BSL065 | нет комментария к экспорту | отключение по CommonCommands | везде кроме форм; **нет** для штатного клиентского `*Завершение`/`*Completion` (оповещение) |
| BSL066 | устаревший метод платформы; особый случай `Предупреждение` | то же по пути | то же по контексту `ОбработкаКоманды` |
| BSL254 | параметры без `Знач` на сервере | только CommonCommands | все серверные процедуры/функции в любом `CommandModule.bsl` |

Параметры стандартного обработчика (`ПараметрКоманды`, `ПараметрыВыполненияКоманды`) по-прежнему исключаются через `_BSL062_SKIP_STANDARD_COMMAND_PARAMS` (имя параметра, не путь).

## Следующие партии

- Правила, где ещё остались regex-трассы при наличии CST для тела метода — выносить в обход `tree-sitter` по мере доработки грамматики.
- Дополнительные эвристики для `none` (нет директивы перед методом) — при необходимости уточнять по справке 1С и типам модулей метаданных.
