# Сверка диагностик с BSLLS (BSL Language Server)

**Источник истины по кодам и приоритетам правил на строке** — реализация в **BSLLS** (JAR из экосистемы 1c-syntax / bsl-language-server). Движок onec-hbk-bsl стремится к тем же каноническим кодам `BSLxxx`, что и сопоставление имён правил BSLLS в `_BSLLS_NAME_TO_CODE` и в [bsl_rules_matrix.md](bsl_rules_matrix.md).

**Язык 1С и слой анализа:** типизация в BSL не является строгой статической (как в Java); регистронезависимость имён и ключевых слов — свойство платформы. Правила вроде **BSL153** (каноническое написание ключевых слов) отражают **стиль/единообразие**, а не «регистр на выполнении».

**CST / AST и эвристики:** **BSL001** строится из узлов **ERROR**/**MISSING** дерева **tree-sitter** (конкретное синтаксическое дерево). Часть правил (BSL002–BSL280) использует **текст строк + процедуры из AST**, отдельные — полноценные обходы CST (`diagnostics_cst.py` для отдельных кодов). Там, где грамматика `tree-sitter-bsl` даёт ложные ERROR на **допустимом** BSL (скобки в многострочном присваивании, условия ``Если (…) …``), в [bsl_parser.py](../src/onec_hbk_bsl/parser/bsl_parser.py) действуют **подавления** для паритета с реальным кодом и BSLLS.

## Воспроизводимость

| Что зафиксировать | Как |
| --- | --- |
| JAR анализатора | Каталог репозитория **`.nosync/bsl-language-server`** (любой `bsl-language-server*.jar` в поддереве) **или** переменная окружения **`BSLLS_JAR`** (путь к одному JAR). |
| JVM | **`BSLLS_JAVA`** — исполняемый файл Java (по умолчанию `java` из `PATH`). |
| Корень workspace | Один и тот же каталог для BSLLS `analyze` и для `onec-hbk-bsl --check` (для CLI укажите **`--workspace`**, иначе используйте общий предок путей к файлам). |
| Профиль правил BSLLS | Файл **`.bsl-language-server.json`** в корне workspace (или эквивалент в IDE), чтобы набор включённых диагностик совпадал между прогонами. |

Если в корне workspace **нет** `.bsl-language-server.json`, BSLLS `analyze` использует **дефолты JAR**.

Сравнение с BSLLS для одного `.bsl` — локально с JAR; отдельных скриптов сравнения в репозитории нет. Пример:

```bash
onec-hbk-bsl --check tests/fixtures/diag_baseline/sample.bsl --workspace /path/to/your/config/root
```

## Намеренные отличия onec-hbk-bsl

Дополнительные правила или иной порог без противоречия BSLLS должны быть отражены в матрице правил. Примеры кодов, дублирующих семантику BSLLS под отдельными номерами и отключённых по умолчанию: **BSL080**, **BSL092**, **BSL107** (см. `DEFAULT_DISABLED` и комментарии в `diagnostics.py`).

**BSL256 / BSL208:** для идентификаторов с двумя алфавитами BSLLS чаще даёт **Typo** (омоглифы), а не **LatinAndCyrillicSymbolInWord**; в `diagnostics.py` это разведено эвристикой `_CYR_HOMOGLYPH_TO_LATIN` / `_mixed_script_identifier_is_homoglyph_typo` (полное совпадение со словарём орфографии BSLLS не гарантируется).

**BSL065 в модуле формы:** для путей `path_is_likely_form_module_bsl` правило **Missing export comment** отключено (как в типичном профиле BSLLS `analyze` для форм).

**BSL153 в модуле формы:** для тех же путей **CanonicalSpellingKeywords** отключено — на `Module.bsl` форм BSLLS часто молчит там, где общий модуль получил бы замечание.

**BSL036 / BSL153:** сложность условия считается по тексту от строки `Если`/`ИначеЕсли` до `Тогда` (включая переносы); на этих строках BSL153 подавляется, если с первой строки условия срабатывает BSL036.

**BSL011 / BSL019:** когнитивная сложность учитывает логические операторы в теле метода (в духе Sonar), чтобы реже расходиться с BSLLS на процедурах с длинными булевыми выражениями.

**BSL046 / BSL199:** при включённых обоих правилах цепочка «Если/ИначеЕсли без Иначе» отдаёт только **BSL199** на строке **КонецЕсли** (как BSLLS `IfElseIfEndsWithElse`), без второго кода BSL046.

**BSL055:** допускается не более одной пустой строки подряд (`MAX_BLANK_LINES=1`), в духе BSLLS `ConsecutiveEmptyLines`.

Правила, завязанные на клиент/сервер, не должны отключаться «по пути» к файлу — см. [CLIENT_SERVER_DIAGNOSTICS.md](CLIENT_SERVER_DIAGNOSTICS.md). На больших модулях возможны точечные расхождения с BSLLS — в первую очередь **BSL256** (Typo); при необходимости зафиксируйте намеренное отличие в матрице правил.

## Цикл согласования с 1С:Предприятие (справка)

При расхождении кодов на строке после локального сравнения с BSLLS: (1) открыть исходник на указанной строке; (2) сверить смысл правила по справке 1С (в т.ч. через MCP **user-1c-help** при наличии); (3) при нехватке формулировок — официальная документация/методички в вебе. Намеренные отличия onec-hbk-bsl фиксируются в [bsl_rules_matrix.md](bsl_rules_matrix.md).
