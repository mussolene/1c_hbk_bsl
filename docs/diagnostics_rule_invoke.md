# Классификация вызова правил диагностики (`RulePhase`)

Модуль `src/onec_hbk_bsl/analysis/diagnostics_rule_registry.py` описывает **как по смыслу вызывается** правило (какие данные ему нужны: строки, CST, процедуры, индекс и т.д.). Это **не** категория BSLLS и **не** замена `RULE_METADATA`: реестр метаданных правил по-прежнему в `diagnostics.py`, а здесь — вспомогательная модель для метрик, бенчмарков и будущей оркестрации (например параллельные фазы).

## Зачем это нужно

- Единое место с **явными** переопределениями для правил с известной формой работы (CST + regex, только строки, индекс метаданных).
- **Эвристика** для остальных кодов по полям `RULE_METADATA` (`tags`, `name`, `sonar_type`), чтобы не дублировать длинные списки вручную.
- Снимок по **включённым** правилам попадает в `DiagnosticEngine.last_metrics["rule_invoke"]` на каждый прогон `check_*` (в том же потоке, что и `last_metrics` в целом).

Движок по-прежнему выполняет правила **одной** последовательностью вызовов в `_run_rules`; модуль реестра **не** подменяет эту цепочку.

## Значения `RulePhase`

| Значение   | Смысл |
|------------|--------|
| `line`     | В основном обход строк / построчные эвристики. |
| `cst`      | Опора на tree-sitter CST при валидном дереве. |
| `proc`     | Нужны границы процедур/методов (`procs`). |
| `region`   | Регионы `#Область` / схожая структура. |
| `index`    | Символьный индекс или метаданные конфигурации (например BSL280). |
| `hybrid`   | Типичный для проекта путь: CST при чистом дереве, иначе regex/строки. |
| `module`   | Уровень файла/заголовка (зарезервировано под расширение эвристики). |
| `other`    | Смешанное или пока не классифицировано эвристикой. |

## Как выбирается фаза

1. **Явная таблица** `_EXPLICIT_PHASE` в `diagnostics_rule_registry.py` (например BSL001 → `cst`, BSL014/BSL119–BSL121 → `line`, BSL280 → `index`, ряд правил с CST+fallback → `hybrid`).
2. Если кода нет в таблице — **эвристика** по `RULE_METADATA[code]`: теги вроде `syntax`, `brain-overload`, `security`, `convention`, имя правила (`ParseError`, `CognitiveComplexity`, …), тип Sonar.
3. Если ничего не подошло — `other` (`source: "default"` в терминах API ниже).

Добавляя новое правило с нестандартной формой вызова, имеет смысл **дополнить** `_EXPLICIT_PHASE`, чтобы классификация не «плыла» от тегов.

## API

- `infer_rule_invoke(code, meta) -> RuleInvokeInfo` — поле `phase` и `source` (`"explicit"` | `"tags"` | `"default"`).
- `build_enabled_invoke_snapshot(engine, RULE_METADATA)` — только правила, для которых `engine._rule_enabled(code)`; результат:
  - `counts_by_phase` — сколько включённых правил в каждой фазе;
  - `codes_by_phase` — отсортированные списки кодов по фазам.
- `parallel_phase_groups` — заготовка для будущего планировщика (какие фазы теоретически можно гонять параллельно при read-only входах); **текущий движок это не использует**.

## Связь с `last_metrics`

Помимо полей про CST/процедуры/регионы (`tree_is_ts`, `proc_source`, …) в `last_metrics` есть ключ **`rule_invoke`** — результат `build_enabled_invoke_snapshot`. Удобно смотреть в бенчах и при отладке профилей `select`/`ignore`.

См. также [cst_policy.md](cst_policy.md) про политику CST и метрики деградации.
